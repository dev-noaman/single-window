# Research: OfficeRnD NestJS BFF

**Date**: 2026-02-08 | **Branch**: `001-officernd-nestjs-bff`

## R1: NestJS Caching Strategy

**Decision**: Use `@nestjs/cache-manager` with in-memory store (default), manual `cacheManager.get/set` in service layer with per-endpoint TTLs.

**Rationale**: The BFF has only 3 endpoints with different caching needs (status: 5s, progress: 2s, sync/run: no cache). Manual cache control via `CACHE_MANAGER` injection gives precise TTL per operation, unlike the global `CacheInterceptor` which applies uniform caching. In-memory store requires zero external dependencies.

**Alternatives considered**:
- Global `CacheInterceptor` with `@CacheTTL` decorators: Simpler but only caches GET routes by URL key; doesn't handle the status endpoint which combines two upstream calls. Manual approach gives full control.
- Redis (`@keyv/redis`): Over-engineered for a single-user admin tool with 3 endpoints. Can be added later via `REDIS_URL` env var.

## R2: Frontend Framework & Static File Serving

**Decision**: Build a new React SPA with Vite inside `officernd/bff/frontend/`. Vite builds to `public/` which NestJS serves via `@nestjs/serve-static` (`ServeStaticModule.forRoot()`). UI is created using the `/frontend-design` skill for a polished, production-grade terminal-style interface.

**Rationale**: The current `index.php` is functional but uses raw Tailwind via CDN and vanilla JS. A React SPA provides component-based architecture (Terminal, StatusBar, ActionButtons), custom hooks for data fetching (useStatus, useSync), and a professional design generated by `/frontend-design`. Vite provides fast builds and HMR during development.

**Alternatives considered**:
- Convert `index.php` to static `index.html` (minimal change): Faster to implement but misses the opportunity for a genuinely improved UI. User explicitly requested a "nice UI" via `/frontend-design`.
- Next.js: Overkill for a single-page admin dashboard with no SSR needs. Vite+React is simpler.
- Vue/Svelte: React is more widely known and has the largest ecosystem for terminal-style UI components.

## R7: PHP Cleanup Strategy

**Decision**: Delete the entire `officernd/web/` directory from the repository.

**Rationale**: Every PHP file is fully replaced by the BFF:
- `index.php` → React SPA served from `public/`
- `status.php` → `GET /api/officernd/status` (NestJS controller)
- `progress.php` → `GET /api/officernd/progress` (NestJS controller)
- `trigger-fetch.php` → `POST /api/officernd/sync/run` (NestJS controller)
- `trigger-increment.php` → `POST /api/officernd/sync/run` (NestJS controller)

Keeping dead PHP files would cause confusion. Clean removal. (Also remove `Dockerfile.web` if present — leftover from an earlier Docker-based setup.)

**Alternatives considered**:
- Keep as archive/backup: Adds clutter; git history preserves everything if needed.

## R3: HTTP Client for Upstream Calls

**Decision**: Use `@nestjs/axios` (`HttpModule` / `HttpService`) for all calls to the FastAPI sync API.

**Rationale**: `@nestjs/axios` is the official NestJS HTTP module. It provides Observable-based responses, configurable timeouts, and integrates with NestJS dependency injection. Axios is already battle-tested for JSON API proxying.

**Alternatives considered**:
- Native `fetch()`: Available in Node 20+ but no NestJS module integration, requires manual timeout handling.
- `got` / `node-fetch`: Viable but not NestJS-idiomatic; would need custom module wrapping.

## R4: Progress Data Source

**Decision**: Proxy to FastAPI `GET /sync/progress` endpoint rather than reading `/tmp/officernd_sync_progress.json` directly.

**Rationale**: Since the BFF runs on the host (not in Docker), it can access the file. However, proxying to the API is cleaner: it decouples the BFF from the file system, and the FastAPI endpoint already reads and returns the same file content. If the FastAPI progress endpoint changes to use a database or in-memory store, the BFF automatically benefits.

**Alternatives considered**:
- Direct file read (`fs.readFileSync`): Faster (no HTTP hop) but couples BFF to file path and OS. Also means the BFF needs shared `/tmp` access, which is fragile.

## R5: Project Location

**Decision**: Place the NestJS app at `officernd/bff/` within the existing officernd directory.

**Rationale**: Keeps all officernd-related code together (`officernd/api/`, `officernd/web/`, `officernd/bff/`). The BFF is logically part of the officernd service family. The name `bff` is concise and descriptive.

**Alternatives considered**:
- `officernd/officernd-bff/`: Redundant nesting.
- Top-level `officernd-bff/`: Breaks the grouping convention used by the existing officernd directory.

## R6: Port Allocation

**Decision**: BFF runs on port 8088, replacing the current PHP dev server on the same port.

**Rationale**: The BFF is a direct replacement for the PHP dev server (`php -S 0.0.0.0:8088`) which currently serves the web UI. All officernd services run on the host: FastAPI at :8000, PHP at :8088, PostgreSQL at :5432. The BFF takes over :8088 after stopping the PHP server.

**Alternatives considered**:
- New port (e.g., 8089): Would require updating nginx config and would leave two UI services for the same feature.
